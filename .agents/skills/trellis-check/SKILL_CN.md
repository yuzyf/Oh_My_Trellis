# 代码质量检查（Code Quality Check）

对最近写出的代码做全面质量核验。综合规范合规、跨层安全与提交前检查。

---

## 步骤 1：识别改了什么

```bash
git diff --name-only HEAD
git status
```

## 步骤 2：阅读任务产物与适用规范

按顺序阅读当前任务产物：

- `prd.md`
- 若有 `design.md`
- 若有 `implement.md`

```bash
python3 ./.trellis/scripts/get_context.py --mode packages
```

对每个变更的包/层，阅读规范索引并跟随其 **Quality Check（质量检查）** 部分：

```bash
cat .trellis/spec/<package>/<layer>/index.md
```

阅读被引用的具体规范文件 —— 索引是指针，不是终点。

## 步骤 3：运行项目检查

运行项目的 lint、类型检查与测试命令。继续前先修掉所有失败。

## 步骤 4：对照清单复查

### 代码质量

- [ ] Linter 通过？
- [ ] 类型检查通过（若适用）？
- [ ] 测试通过？
- [ ] 没有残留调试日志？
- [ ] 没有被压制的警告或类型安全绕过？

### 测试覆盖

- [ ] 新函数 → 加了单元测试？
- [ ] Bug 修复 → 加了回归测试？
- [ ] 行为变更 → 更新了既有测试？

### 规范同步

- [ ] `.trellis/spec/` 是否需要更新？（新模式、约定、经验教训）

> “如果我修了个 bug，或发现了非显而易见的东西，是否该记下来，免得未来的我再踩坑？” → 若 YES，更新相关规范文档。

## 步骤 5：跨层维度（若适用）

若改动只局限在单层，跳过本步。

### A. 数据流（改动触及 3+ 层）

- [ ] 读流追踪正确：Storage → Service → API → UI
- [ ] 写流追踪正确：UI → API → Service → Storage
- [ ] 类型/schema 在层间正确传递？
- [ ] 错误正确向上传播给调用方？

### B. 代码复用（修改常量、创建工具函数）

- [ ] 新建前是否搜索过已有类似代码？
  ```bash
  grep -r "pattern" src/
  ```
- [ ] 若 2+ 处定义同一值 → 是否抽到共享常量？
- [ ] 批量修改后，所有出现点是否都更新了？

### C. 导入/依赖（创建新文件）

- [ ] 导入路径正确（相对 vs 绝对）？
- [ ] 没有循环依赖？

### D. 同层一致性

- [ ] 其他使用同一概念的地方是否一致？

---

## 步骤 6：报告并修复

报告发现的违规并直接修复。修复后重新运行项目检查。

> 译注：阶段 2.2 与提交前最后一轮检查都会落到这里；最后一轮要做全范围，不只看最近那一小段实现。
