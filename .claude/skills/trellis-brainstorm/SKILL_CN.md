# Trellis 头脑风暴（Brainstorm）

## 不可妥协的访谈契约（Interview Contract）

就计划的每一方面对我穷尽式追问，直到形成共同理解。沿设计树的每一分支走下去，逐一消解决策之间的依赖。对每个问题给出你的推荐答案。

一次只问一个问题。

## 不可妥协的证据规则（Evidence Rule）

若问题能通过探索代码库回答，就去探索代码库，不要先问用户。

这是强制要求。在向用户提问前，先检查答案是否已存在于代码、测试、配置、文档、既有规范或任务历史中。

不要让用户确认仓库已经能回答的事实。只问产品意图、偏好、范围、风险承受度，或检查之后仍模糊的决策。

---

在 Phase 1 规划阶段使用本 skill，把用户请求转成清晰需求与规划产物。

## 前置条件（Preconditions）

仅在已获得创建任务的同意、且用户准备进入 Trellis 规划时使用本 skill。

若尚无任务，先创建：

```bash
TASK_DIR=$(python3 ./.trellis/scripts/task.py create "<short task title>" --slug <slug>)
```

标题从用户请求提炼，简洁即可。slug 不要带日期前缀。`task.py create` 会自动给目录加上 `MM-DD-` 前缀。

`task.py create` 会创建默认的 `prd.md`。在追问之前，先用当前理解更新该文件。

## 规划流程（Planning Flow）

1. 把用户请求与已确认事实写入 `prd.md`。
2. 提问前先检查可用证据：
   - 代码、测试、fixtures、配置
   - README、文档、既有 specs、领域笔记
   - 相关 Trellis 任务、research 文件、会话历史（若有）
3. 把发现分成：
   - 已确认事实
   - 仍需用户给出的产品意图
   - 仍需用户拍板的范围或风险决策
   - 可能超出范围的项
4. 只问当前剩余的、价值最高的那一个问题。
5. 提问时附上你的推荐答案。
6. 用户每答一次，先更新 `prd.md` 再继续。
7. 复杂任务在开工前创建或更新 `design.md` 与 `implement.md`。
8. 最终评审或 `task.py start` 之前，跑下面的 PRD 收敛检查。

不要凭空发明项目专属的产品/规范层级。仓库若已有 product、domain 或 spec 文档就用它们；没有就用现有证据推进。

## 提问规则（Question Rules）

每条消息只问一个问题。

每个问题必须包含：

- 需要做的决策
- 为什么这个答案重要
- 你的推荐答案
- 若用户另选的代价/权衡

不要问流程性问题（要不要搜索、要不要看文件、要不要继续 brainstorm）。证据工作直接做。只有剩余问题是产品决策、偏好、范围边界或风险承受度时，才问用户。

## 思考框架：第一性原理分析（First Principles Analysis）

当需求含糊、方案感觉过度设计，或你正要「因为大家都这样」而加复杂度时——先拆到基本事实，再往上推理。

### 步骤 1：重述问题

剥掉实现细节，压成一句话。

> 差：「我们要给用户资料接口加 Redis 缓存」
> 好：「用户资料加载太慢」

### 步骤 2：列出基本事实

什么是绝对为真的（不是意见或惯例）？

| 类别 | 例子 |
|----------|----------|
| **物理约束** | 网络延迟 ≥ 0，磁盘 I/O 有上限 |
| **业务规则** | 「用户必须能看到自己的数据」 |
| **技术不变量** | 「数据必须一致」 |
| **用户需要** | 「用户希望在 Y 秒内得到 X」 |

### 步骤 3：挑战假设

对当前计划的每个组件问：

- **事实还是惯例？**「我们一直用 REST」——为什么？
- **拿掉会怎样？** 若什么都不坏，它就不必要。
- **解决的是真问题还是症状？** 顺着因果链追。
- **谁从这复杂度受益？** 若是「没人」，就简化。

### 步骤 4：从事实往上建

1. 从满足所有事实的最小可行机制开始
2. 仅当某条事实明确要求时才加复杂度
3. 每加一项都要能回答：「哪条事实要求它？」

### 步骤 5：校验

- 方案是否解决了原始问题？
- 哪些假设还需要验证？
- 验证它的最简单实验是什么？

## 产物规则（Artifact Rules）

`prd.md` 记录需求与验收：

- 目标与用户价值
- 已确认事实
- 需求
- 验收标准
- 超出范围
- 仍会阻塞规划的开放问题

`design.md` 为复杂任务记录技术设计：

- 架构与边界
- 数据流与契约
- 兼容与迁移说明
- 重要权衡
- 运维或回滚考量

`implement.md` 为复杂任务记录执行计划：

- 有序实现清单
- 验证命令
- 风险文件或回滚点
- `task.py start` 前的跟进检查

轻量任务可以只有 `prd.md`。复杂任务在 `task.py start` 前必须具备 `prd.md`、`design.md` 与 `implement.md`。

`implement.md` 不能替代 `implement.jsonl`。仅当任务需要清单式规范/研究引用时，才用 JSONL。

## PRD 收敛检查（PRD Convergence Pass）

在宣布规划就绪，或运行 `task.py start` 之前，按上面的产物结构把 `prd.md` 整篇重写一遍。这不是可选清理，而是规划的最终闸门。

本次收敛必须无损：

- 把重复事实压成一个权威小节。
- 把临时 brainstorm 小节（如 `What I already know`、`Assumptions`、已解决的 `Open Questions`）并入 Goal、Background、Requirements、Technical Notes 或 Acceptance Criteria。
- 删掉已解决的开放问题，不要留下空段或已答完的段。
- 当并行的 bug 列表与需求列表描述同一工作时合并它们；每个缺陷的 severity、证据与 file:line 锚点保留在对应需求上。
- 保留每一个 file:line 锚点、决策、约束、需求 ID 与验收标准映射。
- 只保留真正阻塞的开放问题。

收敛后从头到尾读一遍 `prd.md`，确认除非重复能带来新信息，否则事实不跨节重复。

## 质量门槛（Quality Bar）

宣布规划就绪前：

- `prd.md` 含有可测试的验收标准。
- `prd.md` 已通过 PRD 收敛检查：没有未解决的临时 brainstorm 小节、没有跨节重复事实、没有丢失锚点/决策/验收映射。
- 仓库可回答的问题已通过检查回答完毕。
- 剩余开放问题确实关乎用户意图或范围。
- 复杂任务已有 `design.md` 与 `implement.md`。
- 用户已审阅最终规划产物，或明确批准继续。

在用户批准或明确要求实现之前，不要开始实现。
