# 代码质量检查（Code Quality Check）

对最近写出的代码做全面质量核验。综合规范合规、跨层安全与提交前检查。

---

## 步骤 1：识别改了什么

```bash
git diff --name-only HEAD
git status
```

## 步骤 2：阅读任务产物与适用规范

按顺序阅读当前任务产物：

- `prd.md`
- 若有 `design.md`
- 若有 `implement.md`

```bash
python3 ./.trellis/scripts/get_context.py --mode packages
```

对每个变更的包/层，阅读规范索引并遵循其中的 **Quality Check（质量检查）** 小节：

```bash
cat .trellis/spec/<package>/<layer>/index.md
```

阅读索引引用的具体规范文件——索引是指针，不是目标。

## 步骤 3：运行项目检查

运行项目的 lint、类型检查与测试命令。有失败先修，再继续。

## 步骤 4：对照清单复查

### 代码质量（Code Quality）

- [ ] Linter 通过？
- [ ] 类型检查通过（若适用）？
- [ ] 测试通过？
- [ ] 没有残留调试日志？
- [ ] 没有被压制的警告或类型安全旁路？

### 测试覆盖（Test Coverage）

- [ ] 新函数 → 补了单元测试？
- [ ] Bug 修复 → 补了回归测试？
- [ ] 行为变更 → 更新了既有测试？

### 规范同步（Spec Sync）

- [ ] `.trellis/spec/` 是否需要更新？（新模式、约定、经验教训）

> 「如果我修了一个 bug，或发现了不显然的点，要不要写下来，免得以后的自己再踩坑？」→ 若答案是 YES，更新对应规范文档。

## 步骤 5：跨层维度（若适用）

若改动只落在单层，可跳过本步。

### A. 数据流（改动触及 3 层及以上）

- [ ] 读路径可正确追踪：Storage → Service → API → UI
- [ ] 写路径可正确追踪：UI → API → Service → Storage
- [ ] 类型/模式在层间正确传递？
- [ ] 错误正确传播给调用方？

### B. 代码复用（改常量、建工具函数时）

- [ ] 新建前是否搜索过已有相似代码？
  ```bash
  grep -r "pattern" src/
  ```
- [ ] 若 2 处以上定义同一值 → 是否抽到共享常量？
- [ ] 批量修改后，所有出现处是否都更新了？

### C. 导入/依赖（新建文件时）

- [ ] 导入路径正确（相对 vs 绝对）？
- [ ] 没有循环依赖？

### D. 同层一致性

- [ ] 使用同一概念的其他位置是否一致？

---

## 步骤 6：报告并修复

报告发现的违规项并直接修复。修复后重新运行项目检查。
